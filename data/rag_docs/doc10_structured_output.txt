구조화된 출력 제어 (Structured Output)

구조화된 출력 제어는 LLM의 응답을 JSON, XML, YAML 등 특정 형식으로 강제하여 프로그래밍 방식으로 파싱하고 활용할 수 있도록 하는 기법입니다.

방법 1: 프롬프트 지시
"다음 형식의 JSON으로만 응답하세요:
{\"decision\": \"pass|rewrite|escalate\", \"score\": 0~10, \"reasoning\": \"이유\"}"

방법 2: OpenAI response_format 파라미터
```python
response = client.chat.completions.create(
    model="gpt-4o",
    response_format={"type": "json_object"},
    messages=[...]
)
```

방법 3: Pydantic 스키마 활용 (OpenAI Structured Outputs)
```python
from pydantic import BaseModel

class JudgeDecision(BaseModel):
    decision: Literal["pass", "rewrite", "escalate"]
    score: int
    reasoning: str

response = client.beta.chat.completions.parse(
    model="gpt-4o",
    response_format=JudgeDecision,
    messages=[...]
)
```

구조화 출력 설계 원칙:
1. 스키마 명확성: 필드명과 타입을 명확하게 정의
2. 필수/선택 구분: required 필드와 optional 필드를 분리
3. 범위 제약: enum 값이나 범위로 입력 가능한 값을 제한
4. 중첩 구조 최소화: 너무 깊은 중첩은 파싱 오류 유발
5. 오류 처리: JSON 파싱 실패 시 폴백 로직 구현

MOA 파이프라인에서의 활용:
- JudgeAgent: {"decision": "pass", "score": 8, "reasoning": "..."}
- CriticAgent: {"strengths": [...], "weaknesses": [...], "recommendation": "..."}
- RouterAgent: {"path": "moa", "requires_rag": true, "confidence": 0.85}

파싱 실패 방지 전략:
- 응답 시작/끝의 마크다운 코드블록 제거 (```json ... ```)
- try-except로 파싱 오류 처리
- 정규식으로 JSON 부분만 추출
- 재시도 로직 + 오류 메시지를 다음 프롬프트에 포함
