# Study1

## 1. 이 레포지토리는 무엇을 하는가

이 레포지토리는 완성형 AI 제품을 만드는 저장소라기보다, **AI가 일을 더 잘하게 하려면 어떤 부품이 필요하고, 그 부품들을 어떻게 연결해야 하는지 실험하는 랩**에 가깝다.

README 기준 프로젝트 목표는 다음과 같다.

- 단일 LLM 호출 baseline과 MOA 파이프라인의 품질·비용·속도 차이 비교
- Router → Planner → Draft → Critic → Judge → Synthesizer 구조 구현
- 6주차에 RAG·MCP를 통합해 종합 오케스트레이션 검증
- 모든 실행을 trace/logging으로 추적하고 재현 가능하게 유지

즉 이 저장소의 핵심은 **좋은 답변을 만드는 AI 시스템의 구조를 직접 만들어 보고 비교하는 것**이다.

---

## 2. 이 레포를 주방에 비유하면

이 레포는 아직 프랜차이즈나 미슐랭급 완성형 주방은 아니다.

더 정확히 말하면,

- 기본 조리기구는 있음
- 기본 조리 순서도 있음
- 보조 셰프 분업도 있음
- 레시피 참고서(RAG), 외부 도구(MCP)도 붙이기 시작함
- 하지만 상업형 운영 주방처럼 세밀한 자동화·개인화·도메인 특화는 아직 약함

즉 현재 상태는 **요리 가능한 실험실 주방**이다.

---

## 3. 각 부분은 무엇이고 왜 필요한가

### 3-1. Baseline

**무엇인가**

가장 기본형이다. 질문 1개를 모델 1번 호출해서 답을 받는 방식이다.

**왜 필요한가**

비교 기준이 필요하기 때문이다. baseline이 없으면 MOA, RAG, MCP를 붙였을 때 실제로 좋아졌는지 알 수 없다.

**쉽게 말하면**

혼자 바로 문제를 풀어본 기본 점수다.

---

### 3-2. Benchmark

**무엇인가**

AI에게 던질 문제 세트다. 이 레포에서는 `data/benchmarks/v1.json`이 기본 시험지 역할을 한다.

**왜 필요한가**

같은 문제를 써야 결과 비교가 공정해진다.

**쉽게 말하면**

모든 학생에게 같은 시험지를 주는 것이다.

---

### 3-3. MOA 파이프라인

**무엇인가**

답을 한 번에 만들지 않고,

- 여러 draft 생성
- draft 비교와 비판
- 최종 결과 합성
- judge 검토
- 필요 시 rewrite

를 거쳐 결과를 만드는 구조다.

**왜 필요한가**

한 번에 뽑은 답은 빠르지만 실수나 단일 관점 문제가 생길 수 있다. MOA는 여러 단계를 거치며 답을 다듬기 위해 필요하다.

**쉽게 말하면**

혼자 바로 제출하지 않고, 여러 사람이 초안 작성 → 검토 → 최종 정리 → 제출하는 방식이다.

---

### 3-4. Router

**무엇인가**

요청을 보고 어떤 경로로 처리할지 정하는 분기기다.

예:

- 단순한 요청은 `single`
- 복잡한 요청은 `moa`
- 문서 검색이 필요하면 `requires_rag`
- 도구 사용이 필요하면 `requires_mcp`

**왜 필요한가**

모든 질문에 무조건 복잡한 파이프라인을 쓰면 느리고 비싸다. 반대로 모든 질문을 단순 처리하면 품질이 떨어질 수 있다.

**쉽게 말하면**

주방장이 주문을 보고 어느 조리 라인으로 보낼지 정하는 것이다.

---

### 3-5. RAG

**무엇인가**

외부 문서를 참고해서 답하게 하는 구조다.

흐름은 보통 다음과 같다.

1. 문서를 잘게 나눈다
2. 각 조각을 벡터로 바꾼다
3. 벡터 DB에 저장한다
4. 질문이 들어오면 질문도 벡터화한다
5. 가장 가까운 문서 조각을 검색한다
6. 그 조각을 프롬프트에 붙인다
7. LLM이 그 자료를 보고 답한다

**왜 필요한가**

모델 머릿속 지식만으로는 특정 문서, 최신 자료, 근거 기반 답변을 정확히 다루기 어렵기 때문이다.

**쉽게 말하면**

교과서를 펼쳐놓고 그 내용을 참고해 답하는 것이다.

---

### 3-6. MCP

**무엇인가**

모델이 외부 도구를 쓰게 하는 연결 계층이다. 이 레포에서는 현재 filesystem 중심 MCP가 들어가 있다.

예:

- 파일 읽기
- 폴더 목록 보기
- 안전한 whitelist 경로 안에서만 접근

**왜 필요한가**

모델은 원래 파일 시스템을 직접 열거나 외부 도구를 직접 다루지 못한다. 실제 일을 하려면 도구 연결이 필요하다.

**쉽게 말하면**

모델에게 손발과 도구를 붙이는 것이다.

---

### 3-7. Executor

**무엇인가**

전체 파이프라인을 실제로 돌리는 중앙 엔진이다.

- Router 결과를 받음
- 필요하면 RAG 수행
- 필요하면 MCP 수행
- 컨텍스트를 prompt에 합침
- MOA 단계 실행
- 결과와 기록을 남김

**왜 필요한가**

각 부품이 따로 놀면 시스템이 되지 않기 때문이다. Executor가 전체 흐름을 하나로 묶는다.

**쉽게 말하면**

주방의 총괄 조리 시스템이다.

---

### 3-8. Logging / Trace

**무엇인가**

실행 과정 전체를 기록하는 장치다.

예:

- 어떤 경로로 갔는지
- 어떤 에이전트가 호출됐는지
- RAG가 켜졌는지
- MCP가 성공했는지
- 비용과 지연시간이 얼마였는지

**왜 필요한가**

AI 시스템은 결과만 보면 왜 잘됐는지, 왜 실패했는지 알기 어렵다. 기록이 있어야 디버깅과 재현이 가능하다.

**쉽게 말하면**

실험 노트와 조리 기록표다.

---

### 3-9. Evaluation

**무엇인가**

결과 품질을 평가하는 장치다.

기본적으로 다음을 본다.

- clarity
- structure
- constraint_following
- usefulness

경로에 따라 groundedness, citation_traceability, tool_use_correctness 같은 추가 항목도 본다.

**왜 필요한가**

돌아간다고 끝나는 게 아니라, 실제로 좋아졌는지 평가해야 하기 때문이다.

**쉽게 말하면**

맛 평가표다.

---

### 3-10. Comparator

**무엇인가**

결과끼리 비교하는 도구다.

예:

- single vs moa
- moa vs moa+rag
- moa vs moa+mcp

를 그룹 단위로 비교한다.

**왜 필요한가**

RAG나 MCP를 붙였다고 무조건 좋은 게 아니기 때문이다. 품질 향상 대비 비용 증가, 속도 감소를 같이 봐야 한다.

**쉽게 말하면**

공부법 A와 B를 점수·시간·비용으로 비교하는 표다.

---

### 3-11. Tests

**무엇인가**

코드가 의도대로 동작하는지 자동으로 점검하는 코드다.

**왜 필요한가**

AI 프로젝트는 수정했을 때 다른 부분이 쉽게 깨질 수 있다. 테스트가 있어야 변경 후에도 기본 동작이 유지되는지 확인할 수 있다.

**쉽게 말하면**

주방 장비 점검 체크리스트다.

---

## 4. 현재 진행상황

### 이미 구현된 것

- baseline 실행 구조
- benchmark 로딩 구조
- MOA 실행 구조
- Router 기반 분기
- RAG retriever + context 주입
- MCP client + 안전 경로 통제
- trace/logging
- 평가 루브릭
- 결과 비교기
- 일부 테스트 코드

즉 **실험실 주방의 기본 부품과 조리 라인은 이미 구축된 상태**다.

### 아직 약한 것

- 기본 벤치마크가 RAG/MCP를 강하게 트리거하지 않음
- 그래서 `moa+rag`, `moa+mcp`의 실측 증거가 약할 수 있음
- 제품형 UX, 개인화, 장기 memory, 도메인 특화 데이터 운영은 아직 약함
- 이 저장소는 애초에 완성형 제품보다 실험·비교 목적이 강함

즉 지금 단계는:

- **기초 실험실 주방: 거의 완성**
- **RAG/MCP 실증: 중간**
- **제품형 주방: 아직 전 단계**

---

## 5. 왜 이 레포가 중요한가

이 레포의 가치는 “멋진 AI 서비스 완성” 자체보다,

- AI 시스템의 기본 부품이 무엇인지 배우고
- 각 부품이 왜 필요한지 이해하고
- 단순 호출보다 더 나은 구조를 실험하고
- 품질·비용·속도를 비교하는 훈련을 할 수 있다는 점

에 있다.

즉 이 레포는 **AI 오케스트레이션의 기초 공학 실습장**이다.

---

## 6. 초보자가 이 레포를 볼 때 추천 순서

아래 순서로 보면 가장 이해가 쉽다.

### 1단계. README.md

가장 먼저 전체 목표와 구조를 본다.

왜 먼저 보나:
- 이 레포가 제품이 아니라 실험 프로젝트라는 걸 먼저 알아야 함
- 전체 아키텍처를 먼저 봐야 세부 파일이 이해됨

---

### 2단계. data/benchmarks/v1.json

그다음 문제 세트를 본다.

왜 보나:
- 시스템이 무엇을 풀려고 하는지 알아야 함
- baseline과 moa를 무엇으로 비교하는지 이해 가능

---

### 3단계. scripts/run_single.py

가장 단순한 baseline 경로를 먼저 본다.

왜 보나:
- 가장 단순한 흐름을 알아야 이후 복잡한 구조를 이해하기 쉬움
- 입력 → 모델 1회 호출 → 결과 저장의 가장 기본 구조를 익힐 수 있음

---

### 4단계. scripts/run_moa.py

그다음 MOA 전용 실행 스크립트를 본다.

왜 보나:
- baseline과 무엇이 달라졌는지 바로 보임
- 여러 에이전트 단계를 사용하는 이유가 이해됨

---

### 5단계. app/orchestrator/executor.py

이 파일이 핵심이다.

왜 보나:
- RAG와 MCP가 어디서 붙는지 보임
- Draft/Critic/Synthesizer/Judge 흐름이 실제로 어떻게 연결되는지 보임
- 이 레포의 중앙 엔진 역할을 이해할 수 있음

---

### 6단계. app/orchestrator/router.py

그다음 Router를 본다.

왜 보나:
- 어떤 질문이 single, moa, rag, mcp로 가는지 기준을 이해할 수 있음
- 오케스트레이션에서 “분기”가 왜 중요한지 감이 잡힘

---

### 7단계. app/rag/retriever.py

이제 RAG를 본다.

왜 보나:
- 문서를 어떻게 쪼개고 저장하고 검색하는지 이해 가능
- 벡터DB와 retriever가 실제로 무엇을 하는지 구체적으로 보임

---

### 8단계. app/mcp_client/client.py

그다음 MCP를 본다.

왜 보나:
- 파일 읽기나 목록 조회 같은 도구 사용이 어떻게 붙는지 알 수 있음
- whitelist, read-only 같은 안전 장치의 중요성을 이해할 수 있음

---

### 9단계. app/eval/rubric.py

평가 구조를 본다.

왜 보나:
- 단순 실행이 아니라 품질 판단까지 하는 구조를 이해할 수 있음
- 무엇을 기준으로 결과를 좋다고 볼지 알 수 있음

---

### 10단계. scripts/compare_runs.py + app/eval/comparator.py

마지막으로 비교기를 본다.

왜 보나:
- 이 프로젝트가 단순 구현이 아니라 비교 실험용이라는 점이 확실해짐
- 품질 향상 대비 비용과 속도 손해를 함께 보는 사고를 배울 수 있음

---

### 11단계. tests/

끝으로 테스트를 본다.

왜 보나:
- 각 부품이 실제로 어떤 조건에서 동작해야 하는지 더 선명하게 이해 가능
- 초보자에게는 본 코드보다 테스트가 의도를 더 쉽게 보여줄 때가 많음

---

## 7. 핵심 한 줄 정리

이 레포는 **단순히 모델을 여러 개 붙인 저장소가 아니라, AI가 더 잘 일하게 만들기 위해 필요한 부품들(Baseline, Router, MOA, RAG, MCP, Logging, Evaluation, Comparator, Tests)을 하나씩 실험하고 연결해 보는 학습용 오케스트레이션 랩**이다.

그리고 초보자는 반드시

`README → benchmark → run_single → run_moa → executor → router → rag → mcp → eval → comparator → tests`

순서로 보는 것이 가장 이해하기 쉽다.
