레이어 관리 (오케스트레이터/서브에이전트 계층)

멀티 에이전트 시스템에서 레이어 관리는 에이전트들을 계층 구조로 조직하고, 각 계층의 역할과 통신 방식을 정의하는 설계 기법입니다.

계층 구조 유형:

1. 단일 계층(Flat) 구조
- 모든 에이전트가 동일 수준
- 오케스트레이터가 직접 모든 에이전트를 관리
- 장점: 단순, 단점: 확장성 제한

2. 계층적(Hierarchical) 구조
- 오케스트레이터 → 서브오케스트레이터 → 워커 에이전트
- 복잡한 태스크를 계층적으로 분해
- 장점: 확장성, 단점: 통신 오버헤드

3. MOA 파이프라인 구조 (순차+병렬 혼합)
```
Router (라우팅 결정)
    ↓
[Draft_A || Draft_B || Draft_C] (병렬 실행)
    ↓
Critic (초안 분석)
    ↓
Synthesizer (통합)
    ↓
Judge (품질 판정)
    ↓ (rewrite 필요시)
Rewrite (개선) → Judge (재판정)
```

레이어 간 통신 설계:
1. 직렬 통신: A의 출력 → B의 입력 (순차 파이프라인)
2. 병렬 통신: 오케스트레이터가 여러 에이전트에 동시 발송
3. 브로드캐스트: 동일 입력을 모든 에이전트에 전달
4. 집계: 여러 에이전트의 출력을 수집해 하나로 합성

레이어 격리 전략:
- 각 레이어는 자신의 컨텍스트만 접근
- 상위 레이어만이 하위 레이어 에이전트를 직접 호출
- 레이어 간 계약(인터페이스): 입출력 스키마 명확히 정의

레이어별 역할 분리 원칙:
- 라우팅 레이어: 경로 결정, 정책 적용
- 실행 레이어: 실제 LLM 호출, 도구 사용
- 평가 레이어: 품질 판정, 피드백 생성
- 통합 레이어: 결과 병합, 최종 출력 생성

동적 레이어 조정:
요청의 복잡도에 따라 레이어 수를 동적으로 조정:
- 단순 질문 → single path (레이어 최소화)
- 복잡한 분석 → full MOA pipeline (모든 레이어 활성화)
- 긴급 상황 → escalation path (특별 레이어 활성화)

레이어 모니터링:
각 레이어의 지연시간, 토큰 소비, 성공률을 추적하여 병목 지점을 식별하고 최적화합니다.
